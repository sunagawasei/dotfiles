# ルール追加

このスキルは、新しいルールファイルを作成し、特定のファイルパターンに対してClaude Codeが自動的にガイドラインを適用できるようにします。

## ルールとは

- **自動適用**: 指定したファイルパターンに一致するファイルを編集する際に自動的に読み込まれる
- **簡潔**: 50-100行程度の短いガイドライン
- **焦点**: 1つの明確な規約や慣習に集中

## ワークフロー

### 1. ルールのスコープを決定

**質問**: このルールはグローバル（全プロジェクト）ですか、プロジェクト固有ですか？

- **グローバル**: `/Users/s23159/.config/claude/rules/`
  - 例: Go言語規約、セキュリティガイドライン
- **プロジェクト固有**: `<project>/.claude/rules/`
  - 例: プロジェクト固有のコーディング規約、カラーシステム

### 2. 適用対象ファイルの特定

**質問**: "このルールはどのファイルに適用すべきですか？"

ファイルパターンの例：
- `**/*.go` - 全てのGoファイル
- `**/*.{js,ts,jsx,tsx}` - JavaScript/TypeScriptファイル
- `**/test/**` - testディレクトリ内の全ファイル
- `**/config/*.yaml` - configディレクトリ内のYAMLファイル

### 3. Globパターンのガイダンス

#### 基本構文
- `**` - 任意の数のディレクトリ（0個以上）
- `*` - ディレクトリ名/ファイル名内の任意の文字
- `?` - 1文字
- `{a,b}` - aまたはb
- `[abc]` - a、b、cのいずれか

#### 良いパターン例

```yaml
# Go言語ファイル
paths:
  - "**/*.go"
  - "**/go.mod"
  - "**/go.sum"

# フロントエンドファイル
paths:
  - "**/src/**/*.{js,ts,jsx,tsx}"
  - "**/*.css"

# 設定ファイル
paths:
  - "**/.env*"
  - "**/config/**"

# 特定ディレクトリ
paths:
  - "**/nvim/**/*.lua"
```

#### 避けるべきパターン

```yaml
# ❌ 広すぎる
paths:
  - "**/*"  # 全てのファイルに適用される

# ❌ 具体的すぎる
paths:
  - "/Users/s23159/project/src/main.go"  # 絶対パスは避ける
```

### 4. ルール内容の明確化

**質問**: "主なガイドラインや規約は何ですか？"

例：
- "Go言語を優先的に使用する"
- "クレデンシャルを平文表示しない"
- "コミットメッセージはConventional Commits形式"

### 5. Frontmatter生成

```yaml
---
paths:
  - "[パターン1]"
  - "[パターン2]"
trigger: "optional-keyword"  # オプション: 特定の単語で追加トリガー
---
```

**trigger例**:
- `trigger: "commit"` - コミット時に追加で適用
- `trigger: "test"` - テスト時に追加で適用

### 6. ルールファイル作成

#### ファイル名規則
- kebab-case（小文字、ハイフン区切り）
- 説明的で具体的な名前
- 例: `golang.md`, `commit-messages.md`, `security-credentials.md`

#### 基本構造

参照: `references/template.md`

### 7. Frontmatter構文検証

チェック項目：
- [ ] 開始と終了の`---`が存在する
- [ ] `paths:`が配列形式（`-`で始まる）
- [ ] インデントが正しい（スペース2つ）
- [ ] YAML構文が有効

### 8. テスト

作成したルールが正しく動作するか確認：

```bash
# マッチするファイルを確認
find . -path "[パターン]" -type f

# または
echo [パターン]

# 実際にファイルを開いてルールが適用されるか確認
nvim [マッチするファイル]
```

## テンプレート

### 基本テンプレート

```markdown
---
paths:
  - "**/*.ext"
---

# [ルール名]

## 概要

このルールは[目的]を定義します。

## 主要な規約

**[重要なポイント]**

[詳細な説明]

## 例

### 良い例 ✅

```[言語]
[良いコード例]
```

### 悪い例 ❌

```[言語]
[悪いコード例]
```

## 参考

- [関連ドキュメントへのリンク]
```

### trigger付きテンプレート

```markdown
---
paths:
  - "**/*"
trigger: "keyword"
---

# [ルール名]

このルールは[trigger word]を含む操作時に適用されます。

## ガイドライン

[詳細]
```

## 実例

### 実例1: Go言語規約（グローバル）

場所: `/Users/s23159/.config/claude/rules/golang.md`

```yaml
---
paths:
  - "**/*.go"
  - "**/go.mod"
  - "**/go.sum"
---

# Go言語規約

## 言語選択

**コード生成時は、明示的に指定がない限りGolangを使用してください（Pythonは使用しない）**

## プロジェクト構造

- `go.mod`: Goモジュール定義
- `go.sum`: 依存関係のチェックサム
- テストファイル: `*_test.go`パターンに従う
```

### 実例2: コミットメッセージ規約（プロジェクト固有）

場所: `/Users/s23159/.config/.claude/rules/commit-messages.md`

```yaml
---
paths:
  - "**/*"
trigger: "commit"
---

# コミットメッセージ規約

## 基本形式

Conventional Commits形式に従います：

```
<type>(<scope>): <subject>
```

## Type
- `feat`: 新機能追加
- `fix`: バグ修正
- `docs`: ドキュメント変更

## 実例

```
feat(nvim): Snacksターミナル設定を拡張
fix(wezterm): キーバインドの競合を修正
```
```

## トラブルシューティング

### Frontmatter YAMLエラー

```yaml
# ❌ 間違い
---
paths:
- "**/*.go"  # インデントが足りない
---

# ✅ 正しい
---
paths:
  - "**/*.go"  # スペース2つでインデント
---
```

### パターンがマッチしない

```bash
# パターンをテスト
find . -path "**/*.go" -type f | head -5

# glob展開をテスト（zsh）
echo **/*.go
```

### ルールが適用されない

1. Frontmatter構文を確認
2. ファイルパスがパターンにマッチするか確認
3. Claude Codeを再起動

## ベストプラクティス

1. **具体的なパターン**: 必要最小限のファイルにマッチさせる
2. **簡潔に**: 50-100行以内を目標
3. **実例を含める**: 良い例と悪い例を示す
4. **1つの焦点**: 1ルール = 1つの規約
5. **テスト**: 作成後、必ず動作確認
