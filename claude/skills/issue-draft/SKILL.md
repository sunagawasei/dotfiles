# Issue ドラフトワークフロー

コードレビュー結果・Slackスレッド・スクリーンショット・障害報告などを起点に、GitHub Issueを「ドラフト→ユーザー取捨→一括登録→掃除」の4段階で起票する。**いきなり `gh issue create` しない。**

## ワークフロー

### 1. ドラフト作成

- 起点情報からissue候補を抽出する
- `.claude/docs/issue-drafts/` に **1件1ファイル・カテゴリ接頭辞付き**で書き出す
  - 命名例: `struct-01-<slug>.md`, `ui-01-<slug>.md`, `question-01-<slug>.md`, `perf-01-<slug>.md`
- 各ドラフトの構成: タイトル / 本文（簡潔・エンジニア向け。問題点を先に、背景説明は最小限）/ ラベル案
- 起票の根拠（Slack URL・スクショの内容・該当file:line）を本文に含める

### 2. 取捨選択

- ドラフト一覧をユーザーに提示し、keep / drop / 修正 を確認する
- 修正指示があればドラフトを直してから次へ

### 3. 一括登録

- 承認分の登録は **sonnetサブエージェント**（`model: sonnet`）に委譲し、`gh issue create` を一括実行させる
- 登録後、issue番号とURLの一覧を報告する

### 4. 掃除（忘れない）

- **登録済みドラフトはファイル削除する**。`.claude/docs/issue-drafts/` に残骸を残さない
- dropされたドラフトも、ユーザーが「残す」と言わない限り削除する

## 注意

- issue本文は「盛らない」原則に従う（水増し・スコープ外の一般論・選択肢メニューの列挙をしない）
- 誰が発見/実行した事実かを正確に書く。不明ならユーザーに確認
